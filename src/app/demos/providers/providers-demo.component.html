⚙️ Providers — Injeção de Dependência

useValue, useClass, useFactory e useExisting com exemplos práticos.

Como o Angular resolve dependências

O Injetor do Angular mantém um registro de providers. Quando um componente pede inject(MeuServico), o injetor procura o provider mais próximo (componente → módulo → root). As 4 estratégias abaixo controlam o que é entregue.

1. useValue — valor constante

Injeta um valor estático diretamente, sem instanciar uma classe. Use para configurações, constantes e mocks de testes. 

export const APP_CONFIG = new InjectionToken<AppConfig>('app.config');

providers: [
  { provide: APP_CONFIG, useValue: { tema: 'dark', versao: '21.0.0' } }
]

// No componente:
readonly config = inject(APP_CONFIG);
Valor injetado:
  • tema: {{ config.tema }}
  • versão: {{ config.versao }}
  • maxItens: {{ config.maxItens }}

2. useClass — substituir implementação

O componente pede Logger (abstrato), o injetor entrega DevLogger. Em produção, basta trocar por ProdLogger em um só lugar — o código que usa Logger não muda. 

abstract class Logger { abstract log(msg: string): void; }
class DevLogger  extends Logger { log(msg) { console.log('[DEV]', msg); } }
class ProdLogger extends Logger { log(msg) { /* → Sentry */ } }

providers: [
  { provide: Logger, useClass: DevLogger }  // ← troca por ProdLogger em prod
]
@for (m of logMessages; track m) {
{{ m }}
}

3. useFactory — lógica na criação

Uma função é chamada pelo injetor para produzir o valor. Útil quando o valor depende de lógica, outras injeções, ou estado da aplicação. O array deps injeta dependências para a factory. 

providers: [
  {
    provide: DATA_FORMATADA,
    useFactory: () => new Date().toLocaleDateString('pt-BR', {
      weekday: 'long', year: 'numeric', month: 'long', day: 'numeric'
    }),
    // deps: [OutraDependência]  ← injeta no argumento da factory
  }
]
Data formatada pela factory: {{ data }}

4. useExisting — alias / redirecionamento

Cria um alias para um provider existente. Diferente de useClass, não cria uma nova instância — ambos os tokens apontam para o mesmo objeto. Útil para manter retrocompatibilidade ao renomear serviços. 

// Serviço novo e serviço antigo (legacy)
@Injectable() class NovoLogger { ... }
@Injectable() class AntigoLogger { ... }

providers: [
  NovoLogger,
  // AntigoLogger agora é a mesma instância de NovoLogger
  { provide: AntigoLogger, useExisting: NovoLogger }
]

// Quem injeta AntigoLogger recebe NovoLogger — mesma instância!
readonly novo   = inject(NovoLogger);
readonly antigo = inject(AntigoLogger);
// novo === antigo → true
💡 Use useExisting quando múltiplas partes do app injetam tokens diferentes mas precisam da mesma instância (estado compartilhado garantido).
